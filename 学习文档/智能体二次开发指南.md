# OpenClaw 智能体学习与二次开发指南

本文面向“先学透智能体核心，再做二次开发”的目标，基于当前仓库源码整理。

配套启动文档：`学习文档/项目完整启动文档.md`

## 1. 先建立心智模型

OpenClaw 的智能体主链路是：

1. CLI 或 Gateway 收到请求。
2. 解析会话和路由（sessionKey、agentId、delivery 上下文）。
3. 进入统一的 agent 执行入口。
4. `runEmbeddedPiAgent` 串行化执行，完成模型推理 + 工具调用 + 流式事件。
5. 回写会话状态（tokens、模型、report），可选回发消息到渠道。

一句话总结：
OpenClaw 把“入口、会话、模型、工具、流式事件、回写持久化”拆成独立层，二开时只改你需要的那一层。

## 2. 关键源码地图（建议先通读）

### 2.1 入口层

- `src/cli/program/register.agent.ts`
  - 注册 `openclaw agent` 命令。
- `src/commands/agent-via-gateway.ts`
  - 默认通过 gateway 调 `agent`，失败时回退本地 embedded。
- `src/gateway/server-methods/agent.ts`
  - `agent` / `agent.wait` 的网关入口。

### 2.2 执行编排层

- `src/commands/agent.ts`
  - 统一执行入口 `agentCommand` / `agentCommandFromIngress`。
  - 负责 session 解析、技能快照、模型 fallback、事件发射、结果交付。
- `src/commands/agent/session.ts`
  - sessionKey/sessionId 解析与复用策略。
- `src/commands/agent/delivery.ts`
  - 结果打印与渠道投递。

### 2.3 运行时核心层

- `src/agents/pi-embedded-runner/run.ts`
  - 运行队列（session lane + global lane）、重试、fallback、overflow/compaction。
- `src/agents/pi-embedded-runner/run/attempt.ts`
  - 单次 attempt：构建工具、system prompt、会话管理、模型调用。
- `src/agents/pi-embedded-subscribe.ts`
  - 把底层流事件转换成 OpenClaw 的 `assistant/tool/lifecycle` 事件。
- `src/infra/agent-events.ts`
  - 统一事件总线。

### 2.4 工具与策略层

- `src/agents/pi-tools.ts`
  - 汇总 coding tools + OpenClaw tools + policy 过滤。
- `src/agents/openclaw-tools.ts`
  - 注册内置业务工具（sessions、message、nodes、browser、cron 等）。
- `src/agents/tools/*`
  - 每个工具的实现。
- `src/agents/tool-policy.ts`
  - owner-only、allow/deny、plugin 工具分组策略。
- `src/agents/pi-tools.schema.ts`
  - 跨模型 provider 的 schema 兼容清洗。
- `src/agents/schema/typebox.ts`
  - 工具 schema 的枚举封装（避免 `anyOf`/`Type.Union` 坑）。

### 2.5 模型与多智能体层

- `src/agents/model-selection.ts`
  - provider/model 规范化、alias、allowlist。
- `src/agents/model-fallback.ts`
  - fallback 候选与执行。
- `src/agents/agent-scope.ts`
  - 多 agent 的 workspace/agentDir/skills 作用域解析。
- `src/agents/subagent-spawn.ts`
  - `sessions_spawn` 的核心执行与深度控制。
- `src/agents/tools/sessions-spawn-tool.ts`
  - `sessions_spawn` 工具参数与调度。

## 3. 学习顺序（建议）

1. 先跑通命令链路（CLI -> Gateway -> Runtime）。
2. 再看 session 路由与 store 回写。
3. 再看工具装配与策略。
4. 最后看 subagent / sandbox / fallback 等高级能力。

推荐阅读顺序：

1. `src/commands/agent-via-gateway.ts`
2. `src/gateway/server-methods/agent.ts`
3. `src/commands/agent.ts`
4. `src/agents/pi-embedded-runner/run.ts`
5. `src/agents/pi-embedded-runner/run/attempt.ts`
6. `src/agents/pi-tools.ts`
7. `src/agents/openclaw-tools.ts`
8. `src/agents/subagent-spawn.ts`

## 5. 二次开发场景与改动入口

### 场景 A：新增一个内置工具

改动路径：

1. 在 `src/agents/tools/` 新建工具实现（建议与现有 `*-tool.ts` 风格一致）。
2. 在 `src/agents/openclaw-tools.ts` 注册工具。
3. 如需配置项，补 `src/config/types.tools.ts` 和对应 zod schema。
4. 如需策略控制，补 `src/agents/tool-policy.ts` 或 policy pipeline。
5. 补测试：
   - 工具单测放 `src/agents/tools/*.test.ts`
   - 必要时补 `src/agents/pi-tools*.test.ts`

注意：工具参数 schema 尽量保持顶层 `type: "object"`，避免复杂 union；优先用 `optionalStringEnum`。

### 场景 B：调整智能体模型选择或 fallback

改动路径：

- `src/agents/model-selection.ts`
- `src/agents/model-fallback.ts`
- `src/agents/pi-embedded-runner/run.ts`

重点：

- 不要只改“模型字符串”，要同时确认 allowlist、fallback、auth profile cooldown 行为。
- 变更后至少验证：普通成功路径 + fallback 路径 + timeout 路径。

### 场景 C：调整会话路由和上下文隔离

改动路径：

- `src/commands/agent/session.ts`
- `src/routing/session-key.ts`
- `src/sessions/session-key-utils.ts`
- `src/config/sessions/*`

重点：

- sessionKey 结构是系统级主键，改动必须检查多渠道兼容。
- 重点回归 direct/group/topic/thread 的 key 生成与复用。

### 场景 D：扩展子智能体（sessions_spawn）能力

改动路径：

- `src/agents/tools/sessions-spawn-tool.ts`
- `src/agents/subagent-spawn.ts`
- `src/agents/tools/sessions-access.ts`

重点：

- 深度限制、并发限制、sandbox 继承、thread binding 是四个核心安全点。
- 任何子智能体能力提升，都要重新审查工具权限边界。

### 场景 E：技能系统增强（skills）

改动路径：

- `src/agents/skills/workspace.ts`
- `src/agents/skills.ts`
- `src/gateway/server-methods/skills.ts`
- `src/config/types.skills.ts`

重点：

- 技能有快照机制（新 session 生效），注意“改了配置但旧会话不生效”是预期行为。
- 三层来源优先级：workspace > managed > bundled。

## 6. 配置层重点（与智能体最相关）

最常改的配置块：

- `agents.defaults.*`
  - `model` / `models`
  - `workspace`
  - `timeoutSeconds`
  - `subagents.*`
  - `sandbox.*`
- `agents.list[].*`
  - agent 级覆盖（workspace、model、skills、tools、sandbox）
- `tools.*`
  - `exec`、`sessions.visibility`、`subagents.tools`、`sandbox.tools`
- `skills.*`
  - `entries`、`load`、`limits`

类型定义入口：

- `src/config/types.agent-defaults.ts`
- `src/config/types.agents.ts`
- `src/config/types.tools.ts`
- `src/config/types.skills.ts`

## 7. 调试建议（高频有效）

1. 先看生命周期是否完整：是否有 `lifecycle start -> end/error`。
2. 再看当前 run 的 sessionKey 是否符合预期。
3. 再看工具是否被 policy 过滤（不是“工具坏了”，而是“工具没被放出来”）。
4. 再看模型侧问题（auth/rate limit/timeout/context overflow）。
5. 最后看 delivery 问题（渠道/target/account/thread）。

对应排查文件：

- 事件：`src/infra/agent-events.ts`
- 等待与去重：`src/gateway/server-methods/agent-job.ts`、`src/gateway/server-methods/agent.ts`
- 工具装配：`src/agents/pi-tools.ts`
- 结果交付：`src/commands/agent/delivery.ts`

## 8. 二开落地顺序（建议照做）

1. 先写最小改动版本（只改一个入口文件）。
2. 立刻补对应单测。
3. 再做策略与配置层接入。
4. 最后补文档与 e2e。

这样做可以避免一次改太多层，导致你无法定位回归来源。

## 9. 你可以直接从这里开始

如果你现在要开始“智能体二开”，推荐第一个可控任务：

- 新增一个最小内置工具（只返回结构化文本）
- 让它在 `openclaw agent` 中被调用
- 为它补 2 个测试：正常输入、非法输入

这条路径最短，能帮你把“工具注册 -> 运行时 -> 事件 -> 回包”完整跑通。
